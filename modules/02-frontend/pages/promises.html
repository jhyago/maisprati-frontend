JavaScript no Navegador

Promises & Assincronia

Algumas tarefas demoram: buscar dados na rede, ler um arquivo, esperar um temporizador. O JavaScript não fica parado esperando — ele segue em frente e te entrega uma Promise: a promessa de um resultado que chega no futuro. Entender Promises é o que destrava o fetch, o async/await e quase toda API moderna.

01

Síncrono vs assíncrono

por que precisamos disso

JavaScript roda em uma única thread: uma linha por vez, de cima para baixo. Se uma tarefa demorada bloqueasse essa thread, a página inteira congelaria — nada de rolar, clicar ou digitar. A solução é o código assíncrono: a tarefa demorada roda "ao lado" e avisa quando termina, deixando a página responsiva enquanto isso.

a ordem surpreende quem está começando 
console.log('1 — agora');
setTimeout(() => console.log('2 — depois (assíncrono)'), 0);
console.log('3 — agora também');

// Saída: 1, 3, 2
// Mesmo com 0ms, o setTimeout só roda quando a thread principal esvazia.
📌 Callback: o jeito antigo Antes das Promises, passava-se uma função de retorno (callback) para rodar "quando terminar". Aninhar muitos callbacks gerava o temido callback hell — código em forma de pirâmide, difícil de ler e de tratar erros. As Promises nasceram para resolver exatamente isso.
02

Os três estados de uma Promise

pending · fulfilled · rejected

Uma Promise é um objeto que representa um valor ainda não disponível. Ela vive em um de três estados — e, uma vez que "se decide" (settled), nunca mais muda:

⏳ pending
pendente — ainda esperando o resultado
✓ fulfilled
resolvida com sucesso (tem um valor)
✕ rejected
falhou (tem um motivo/erro)
o ciclo de vida 
          ┌──────────────► fulfilled  (resolve(valor))
 pending ─┤
          └──────────────► rejected   (reject(erro))

  estado inicial            estado final (settled) — imutável

03

Criando uma Promise

new Promise(resolve, reject)

Você raramente cria Promises na mão (o fetch já devolve uma), mas entender o construtor revela como tudo funciona. Ele recebe uma função com dois "botões": resolve (deu certo) e reject (deu errado).

uma Promise do zero 
function esperar(ms) {
  return new Promise((resolve, reject) => {
    if (ms < 0) {
      reject(new Error('tempo inválido'));   // caminho de falha
      return;
    }
    setTimeout(() => resolve(`esperei ${ms}ms`), ms);  // caminho de sucesso
  });
}

esperar(500).then(msg => console.log(msg));   // "esperei 500ms"
💡 resolve e reject só contam uma vez Depois que a Promise se decide, chamadas seguintes a resolve/reject são ignoradas. Isso garante que um resultado é sempre definitivo.
04

Consumindo: .then / .catch / .finally

reagir ao resultado

Para usar o valor de uma Promise, você se "inscreve" nela. Cada método trata um momento diferente:

MétodoRoda quando…
.then(valor => …)a Promise resolve (sucesso)
.catch(erro => …)a Promise rejeita (erro)
.finally(() => …)sempre, no fim — deu certo ou não
encadeando 
esperar(300)
  .then(msg => {
    console.log('sucesso:', msg);
    return msg.length;          // o que você retorna vira o valor do PRÓXIMO .then
  })
  .then(tamanho => console.log('tamanho:', tamanho))
  .catch(erro => console.log('falhou:', erro.message))
  .finally(() => console.log('acabou (sempre roda)'));
⚠️ O encadeamento é a mágica O que você return dentro de um .then vira o valor do próximo. E se retornar outra Promise, a cadeia espera ela resolver antes de seguir. Foi assim que as Promises mataram o callback hell: pirâmide vira lista achatada de .then.
05

async / await — a forma moderna

Promises com cara de código normal

async/await é "açúcar sintático" sobre Promises: o mesmo motor por baixo, mas escrito como código síncrono, de cima para baixo. Uma função async sempre devolve uma Promise; o await "pausa" a função até a Promise resolver e te dá o valor direto.

o mesmo código, dois estilos 
// Com .then()
function versaoThen() {
  return esperar(200)
    .then(msg => console.log(msg))
    .catch(err => console.log(err.message));
}

// Com async/await — lê como texto comum
async function versaoAsync() {
  try {
    const msg = await esperar(200);   // "pausa" aqui até resolver
    console.log(msg);
  } catch (err) {
    console.log(err.message);         // o reject vira exceção no catch
  }
}
🚨 await só funciona dentro de async Esquecer o async na função (ou usar await no topo de um script antigo) gera erro de sintaxe. E lembre: o reject de uma Promise vira uma exceção — por isso o try/catch substitui o .catch.
06

Várias Promises ao mesmo tempo

Promise.all · race · allSettled

Quando você tem várias tarefas assíncronas, raramente quer esperá-las uma por uma. Os combinadores rodam tudo em paralelo e juntam os resultados:

CombinadorResolve quando…Uso típico
Promise.all([…])todas resolverem (rejeita se uma falhar)buscar 3 endpoints e só seguir com todos
Promise.allSettled([…])todas terminarem (sucesso ou falha)quero o resultado de cada, mesmo se algumas falharem
Promise.race([…])a primeira terminar (boa ou má)timeout: "o que vier primeiro"
Promise.any([…])a primeira que resolver com sucessovários espelhos, vale o mais rápido que der certo
paralelo com Promise.all 
async function carregarTudo() {
  // As três disparam JUNTAS — o tempo total é o da mais lenta, não a soma
  const [a, b, c] = await Promise.all([
    esperar(300),
    esperar(500),
    esperar(200),
  ]);
  console.log(a, '|', b, '|', c);
}
💡 Paralelo > sequencial Dois await seguidos rodam um depois do outro (somam o tempo). Se as tarefas são independentes, dispare-as juntas com Promise.all — bem mais rápido.

Pratique

fixe o conteúdo

Pense na resposta antes de revelar.

1Qual a ordem? ◆ Fácil

O que imprime console.log('A'); Promise.resolve().then(() => console.log('B')); console.log('C');?

A, C, B. Os console.log síncronos rodam primeiro (A, C). O .then é assíncrono: entra na fila de microtarefas e só roda quando a thread principal esvazia — por isso B vem por último.

2Converta para async/await ◆ Médio

Reescreva usando async/await e try/catch:

solução
async function pegar() {
  try {
    const msg = await esperar(100);
    console.log('ok:', msg);
  } catch (e) {
    console.log('erro:', e.message);
  }
}
3all ou allSettled? ◆ Difícil

Você dispara 5 requisições e quer o resultado de cada uma, mesmo que algumas falhem — sem que uma falha derrube as outras. Qual combinador usar, e por quê não o Promise.all?

Promise.allSettled. O Promise.all rejeita inteiro assim que uma Promise falha (você perde os resultados das que deram certo). Já o allSettled espera todas terminarem e devolve um array com { status, value } ou { status, reason } para cada uma.
