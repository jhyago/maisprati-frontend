JavaScript no Navegador

Web Components

E se você pudesse criar suas próprias tags HTML — como <user-card> ou <click-counter> — reutilizáveis em qualquer página, com estilo isolado que nunca vaza? Isso é Web Components: um padrão nativo do navegador (sem framework) feito de três pilares — Custom Elements, Shadow DOM e <template>.

01

Os três pilares

a base do padrão

"Web Components" é um nome guarda-chuva para três tecnologias que trabalham juntas. Você pode usar uma sem as outras, mas combinadas elas entregam componentes encapsulados de verdade.

Custom Elements
define tags novas com comportamento próprio
Shadow DOM
isola o HTML e o CSS internos do componente
<template> & <slot>
molde de marcação reaproveitável + encaixes
📌 Por que isso importa? O grande problema do frontend é o vazamento: um .botao do seu CSS pode, sem querer, estilizar o .botao de outra parte do site. Web Components resolvem isso de forma nativa — o componente vira uma "caixa-preta" com fronteiras reais.
02

Custom Elements

class extends HTMLElement

Um Custom Element é uma classe que estende HTMLElement e é registrada com um nome. A única regra do nome: precisa ter um hífen (click-counter, não counter) — assim o navegador nunca confunde com uma tag nativa futura.

um elemento mínimo 
class SaudacaoPrati extends HTMLElement {
  // Chamado quando o elemento entra na página:
  connectedCallback() {
    this.innerHTML = `<strong>Olá da +PraTi! 👋</strong>`;
  }
}

// Registra a tag <saudacao-prati> (precisa do hífen!)
customElements.define('saudacao-prati', SaudacaoPrati);
…e usar como qualquer tag 
<saudacao-prati></saudacao-prati>

A classe tem métodos de ciclo de vida que o navegador chama automaticamente:

CallbackQuando dispara
constructor()Na criação do elemento (antes de entrar na página)
connectedCallback()Quando o elemento é inserido no DOM — ótimo para montar conteúdo
disconnectedCallback()Quando é removido do DOM — ótimo para limpeza
attributeChangedCallback()Quando um atributo observado muda
03

Shadow DOM — o encapsulamento

attachShadow

O Shadow DOM dá ao componente uma árvore privada. O CSS que você escreve lá dentro não escapa para a página, e o CSS da página não entra. É a fronteira que faz o componente ser realmente reutilizável.

com Shadow DOM 
class CartaoPrati extends HTMLElement {
  constructor() {
    super();
    // Cria a árvore-sombra (modo "open" permite inspecioná-la):
    this.attachShadow({ mode: 'open' });
  }

  connectedCallback() {
    this.shadowRoot.innerHTML = `
      <style>
        /* Este CSS fica PRESO aqui dentro — não vaza! */
        .box { padding: 14px; border-radius: 10px; background: #0080ED; color: #fff; }
      </style>
      <div class="box">Estou isolado do resto da página.</div>
    `;
  }
}
customElements.define('cartao-prati', CartaoPrati);
💡 :host estiliza o próprio componente Dentro do Shadow DOM, o seletor :host aponta para a tag do componente em si (<cartao-prati>). Use-o para definir display, margens e o "exterior" do componente.
04

<template> e <slot>

molde + encaixes

Um <template> é um pedaço de HTML que o navegador não renderiza — fica guardado, pronto para ser clonado quantas vezes você quiser. Já o <slot> é um "buraco" onde o conteúdo escrito entre as tags do componente é encaixado.

slot: conteúdo de fora entra no componente 
<!-- No Shadow DOM do componente: -->
<p>Título fixo</p>
<slot>Texto padrão se nada for passado</slot>

<!-- No uso: "Olá!" cai dentro do <slot> -->
<meu-cartao>Olá!</meu-cartao>
📦 <template> + cloneNode Em vez de montar uma string de HTML toda vez, defina o molde uma vez num <template> e use template.content.cloneNode(true) a cada instância. É mais rápido e mantém a marcação fora do JavaScript.
05

Exemplo completo: <click-counter>

tudo junto, funcionando

Vamos juntar os três pilares num componente real. Ele tem estado próprio (a contagem), estilo isolado via Shadow DOM, e um atributo label configurável. As três instâncias abaixo são de verdade e cada uma tem seu próprio contador — clique nelas:

componentes ao vivo
click-counter.js — o código real desta página 
class ClickCounter extends HTMLElement {
  constructor() {
    super();
    this.count = 0;
    this.attachShadow({ mode: 'open' });
  }

  connectedCallback() {
    const label = this.getAttribute('label') || 'Cliques';
    this.shadowRoot.innerHTML = `
      <style>
        button { font: inherit; cursor: pointer; padding: 10px 16px;
                 border: none; border-radius: 10px;
                 background: #0080ED; color: #fff; font-weight: 600; }
        button:hover { background: #0a6fd0; }
        .n { background: rgba(255,255,255,.25); border-radius: 6px;
             padding: 1px 8px; margin-left: 8px; }
      </style>
      <button>${label} <span class="n">0</span></button>
    `;
    const btn = this.shadowRoot.querySelector('button');
    const n = this.shadowRoot.querySelector('.n');
    btn.addEventListener('click', () => { this.count++; n.textContent = this.count; });
  }
}
customElements.define('click-counter', ClickCounter);
💡 Note o isolamento O CSS button { … } dentro do componente é genérico de propósito — e mesmo assim não afeta nenhum outro botão desta página (como o "Ver solução" abaixo). É o Shadow DOM trabalhando.
06

Pratique

fixe o conteúdo

Pense na resposta antes de revelar.

1O nome inválido ◆ Fácil

Por que customElements.define('counter', Counter) dá erro, mas customElements.define('click-counter', Counter) funciona?

O nome de um Custom Element precisa ter um hífen. É a regra que garante que ele nunca colida com uma tag nativa do HTML (que nunca têm hífen). counter é inválido; click-counter é válido.

2Onde montar o conteúdo? ◆ Médio

Você quer ler um atributo da tag e renderizar o HTML do componente com base nele. Em qual callback do ciclo de vida esse código deve ficar — constructor ou connectedCallback? Por quê?

No connectedCallback. No constructor o elemento ainda não está no DOM e nem sempre seus atributos/filhos estão disponíveis; mexer no conteúdo ali é desencorajado. O connectedCallback roda quando o elemento já entrou na página — é o lugar certo para ler atributos e montar a UI.

3CSS que não vaza ◆ Difícil

Um colega reclama que o p { color: red } definido dentro do componente dele está deixando todos os parágrafos da página vermelhos. O que provavelmente faltou, e como resolver?

Faltou o Shadow DOM. Sem attachShadow, o <style> do componente vive no DOM normal e vaza para a página inteira. A correção é criar a árvore-sombra no construtor e renderizar dentro dela:

resposta_3
constructor() {
  super();
  this.attachShadow({ mode: 'open' });   // cria a fronteira
}
connectedCallback() {
  this.shadowRoot.innerHTML = `<style>p { color: red }</style><p>…</p>`;
}
