JavaScript no Navegador

Consumo de APIs

Sua página raramente vive sozinha — ela busca dados de servidores: o clima, um CEP, uma lista de produtos. Esses servidores expõem APIs, e o JavaScript conversa com elas via fetch pela rede, recebendo JSON de volta. Aqui você aprende o fluxo completo e o coroa consultando o ViaCEP de verdade.

01

API, HTTP e JSON

o vocabulário

Uma API (Application Programming Interface) é um "balcão de atendimento" de um servidor: você faz um pedido (request) para uma URL e recebe uma resposta (response). A conversa segue o protocolo HTTP, e os dados quase sempre voltam em JSON — um texto com a mesma cara de um objeto JavaScript.

Método HTTPIntenção
GETBuscar/ler dados (não muda nada no servidor)
POSTCriar um novo recurso
PUT / PATCHAtualizar um recurso existente
DELETERemover um recurso
📌 Códigos de status Toda resposta traz um número que resume o que aconteceu: 2xx = deu certo (ex.: 200 OK), 4xx = erro do cliente (ex.: 404 Não encontrado), 5xx = erro no servidor. Você vai checar isso no código.
02

fetch + async/await

a requisição

O fetch(url) dispara a requisição e devolve uma Promise — uma "promessa" de resposta que chega no futuro, porque a rede leva tempo. A forma mais legível de esperar por ela é async/await: o await "pausa" a função até a promessa resolver.

GET com async/await 
async function buscarUsuario() {
  const resposta = await fetch('https://api.exemplo.com/users/1');  // 1) requisição
  const dados = await resposta.json();   // 2) lê o corpo e converte JSON → objeto
  console.log(dados.nome);               // 3) usa os dados
}
buscarUsuario();

São dois await: um para a resposta chegar, outro para ler o corpo. O mesmo código com .then() seria:

a mesma coisa com .then() 
fetch('https://api.exemplo.com/users/1')
  .then(resposta => resposta.json())
  .then(dados => console.log(dados.nome));
💡 Prefira async/await Os dois fazem o mesmo, mas async/await lê de cima para baixo, como código normal, e combina com try/catch para tratar erros. É o estilo recomendado hoje.
03

Tratando erros

try/catch · response.ok

A rede falha: o servidor pode estar fora, o usuário sem internet, a URL errada. Código de API sempre precisa prever isso. Há duas armadilhas clássicas que pegam todo iniciante:

🚨 fetch NÃO rejeita em 404/500 Surpresa: para o fetch, receber um 404 ainda é uma resposta que "chegou com sucesso". Ele só cai no catch em falha de rede. Para tratar status de erro, você precisa checar resposta.ok (que é true só para 2xx) você mesmo.
o padrão robusto 
async function buscar(url) {
  try {
    const resposta = await fetch(url);

    if (!resposta.ok) {                        // checa 4xx/5xx manualmente
      throw new Error('HTTP ' + resposta.status);
    }

    const dados = await resposta.json();
    return dados;
  } catch (erro) {
    console.log('Falhou:', erro.message);      // rede caiu OU status ruim
    return null;
  }
}
04

Exemplo real: ViaCEP

fetch de verdade

O ViaCEP é uma API pública e gratuita que, dado um CEP, devolve o endereço. O formato da URL é simples: https://viacep.com.br/ws/{cep}/json/. Vamos usá-la para autopreencher um formulário de endereço — o caso de uso mais comum em checkout de e-commerce.

busca de CEP ao vivo (requisição real à internet)

Experimente: 01001-000 · 20040-002 · 30130-010

buscarCep.js — o código real desta página 
async function buscarCep(cep) {
  // 1) Limpa: o ViaCEP quer só os 8 dígitos
  const limpo = cep.replace(/\D/g, '');
  if (limpo.length !== 8) throw new Error('CEP deve ter 8 dígitos');

  // 2) Requisição
  const resposta = await fetch(`https://viacep.com.br/ws/${limpo}/json/`);
  if (!resposta.ok) throw new Error('Erro de rede: ' + resposta.status);

  // 3) Converte JSON → objeto
  const dados = await resposta.json();

  // 4) Pegadinha do ViaCEP: CEP inexistente NÃO dá 404 —
  //    vem 200 com { "erro": true }. Trate isso!
  if (dados.erro) throw new Error('CEP não encontrado');

  return dados;   // { logradouro, bairro, localidade, uf, ... }
}
⚠️ Duas pegadinhas do ViaCEP (1) Ele espera o CEP só com números — tire o hífen antes. (2) Um CEP que não existe volta com status 200 e corpo { "erro": true }, e não um 404. Por isso, além do resposta.ok, você precisa checar dados.erro.
05

Enviando dados (POST)

além do GET

Para criar ou enviar algo (um cadastro, um comentário), use POST. Aí o fetch ganha um segundo argumento de opções: o método, os cabeçalhos e o corpo (que vai como string JSON).

POST com corpo JSON 
const resposta = await fetch('https://api.exemplo.com/usuarios', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },   // "estou enviando JSON"
  body: JSON.stringify({ nome: 'Ana', email: 'ana@mail.com' }),
});
const criado = await resposta.json();
console.log('Criado com id', criado.id);
📌 body é sempre texto O corpo da requisição precisa ser uma string — por isso o JSON.stringify. E o cabeçalho Content-Type: application/json avisa o servidor de como interpretar esse texto.
06

Pratique

fixe o conteúdo

Pense na resposta antes de revelar.

1Os dois await ◆ Fácil

No fluxo const r = await fetch(url); const d = await r.json();, por que são necessários dois await?

O primeiro espera a resposta chegar pela rede (cabeçalhos, status). O segundo espera o corpo ser lido e convertido de JSON para objeto — r.json() também é assíncrono porque o corpo pode chegar em pedaços.

2O 404 silencioso ◆ Médio

Um colega diz: "meu try/catch envolve o fetch, então qualquer erro da API cai no catch". A afirmação está correta? Justifique.

Não. O fetch só rejeita (cai no catch) em falha de rede. Um 404 ou 500 é considerado uma resposta bem-sucedida pelo fetch. Para tratá-los, é preciso checar resposta.ok e lançar o erro manualmente com throw.

3CEP "válido" que falhou ◆ Difícil

Você busca 99999-999 no ViaCEP. A resposta vem com status 200 e resposta.ok é true — mas o endereço não existe. Como seu código detecta esse caso?

O ViaCEP devolve 200 com o corpo { "erro": true } para CEPs inexistentes. Como o status é "ok", resposta.ok não ajuda — é preciso inspecionar o corpo:

resposta_3
const dados = await resposta.json();
if (dados.erro) throw new Error('CEP não encontrado');
